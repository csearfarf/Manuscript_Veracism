dynamic application scan (DAST) using Burp Suite, followed by manual
verification of findings and targeted penetration testing.  Five confirmed
vulnerabilities were identified and subsequently remediated before the initial
controlled deployment.  The tables below summarise the findings, affected
endpoints, and remediation status.

  \label{tab:perf-observations}
\end{table}

\noindent The performance testing exercise revealed that the system infrastructure
had a rate limiting policy configured at the Nginx reverse proxy layer that
was set below the threshold required for the anticipated 100-concurrent-user
load.  Version~1.0 results showed a notable proportion of failed requests
attributed to HTTP~429 (Too Many Requests) responses returned by the proxy
before requests reached the application layer.  Version~1.1 results, obtained
after disabling the rate limiting rules for the test, demonstrated that the
application itself handled the concurrent load with substantially lower error
rates, confirming that the underlying .NET~8 application server and the
SQL~Server database can sustain the tested load.  The rate limiting
configuration requires re-tuning before production release to balance
protection against abuse with legitimate concurrent usage.
