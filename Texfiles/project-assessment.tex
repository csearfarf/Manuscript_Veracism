\section{Project Assessment}

This chapter describes how VERACISM will be evaluated once development is
complete. The goal of the assessment process is to confirm that the system is
functional, secure, usable, and ready for deployment within the International
School Manila (ISM) environment. For the proposal stage, the focus is on the
planned testing approach: the types of tests to be performed, the tools to be
used, and the metrics that will be collected. Actual test results will be
produced only after implementation.

\subsection{Functional Testing}

Functional testing focused on verifying that the implemented features of
VERACISM behaved according to the intended system requirements. The testing
scope covered the main document-processing functions of the platform,
including file upload, validation, malware scanning, hashing, CID generation,
metadata retrieval, controlled download, dashboard and audit visibility,
tenant-scoped access, and public verification.

The functional assessment used a \textbf{hybrid approach} composed of
\textbf{automated black-box API testing} and \textbf{manual role-based
functional testing}. Automated execution was carried out through Postman
collections run using Newman in the staging environment. These automated
tests covered the public API v1 workflow and selected end-to-end user
journeys, including IT Office upload, Registrar upload, student and external
verification, protected download behaviour, and cross-role upload-to-verify
flows.

Because the current Newman suite primarily exercises the public API v1
surface, manual functional testing was also included for modules and flows
that require user interface interaction, Bearer JWT protected internal
endpoints, QR-based testing with a mobile device, dashboard inspection,
tenant-scoped access checks, and role-based administration tasks.

Manual functional procedures were prepared and executed for the System
Administrator, IT Office / tenant user, Administration Office / Registrar,
Student, External Auditor / Educator, and cross-role end-to-end scenarios.
These procedures covered tenant approval, user and role management, API key
and configuration changes, audit log review and export, health monitoring,
QR display and scan validation, dashboard status review, report deletion or
immutability checks, comprehension checks, cross-tenant access boundary
tests, and multi-network verification checks.

The primary criteria used in functional testing included correct HTTP status
codes, correct response structures, presence and format of expected fields,
correct role and tenant access enforcement, correct UI behaviour where
applicable, successful completion of required user journeys, and retention of
the required audit evidence. Supporting automated and manual functional
testing records will be retained as appendix evidence.

\subsubsection{Functional test success criteria}

To make the functional assessment explicit and reproducible, each test area is
defined with measurable criteria for a \textit{Pass} result. These criteria
focus on correct outputs, correct security behaviour, correct logging,
observable handling of failures, and coverage of both automated and manual
functional modules.

\subsection{Integration Testing}

Integration testing focused on verifying that the major VERACISM components
worked correctly when combined into a complete operational pipeline. The
integration scope covered the interaction between the .NET 8 backend API,
SQL Server, ClamAV, YARA, Lighthouse/IPFS storage, and the public
verification process.

The integration assessment was performed through automated API-based tests in
the staging environment. These tests targeted the public API v1 endpoints
through API key authentication and covered the main integrated operations of
the platform, including upload and scan processing, report retrieval by CID,
public verification, controlled file download, and storage-status checking.
Both valid and invalid cases were included to assess API contract behaviour,
error handling, response consistency, and subsystem coordination.

The integration design also checked whether accepted uploads moved correctly
through the scanning, hashing, metadata, and storage stages, and whether
invalid or unauthorized requests were blocked without creating unintended
records. Additional attention was given to tenant isolation,
public-versus-protected endpoint behaviour, and consistency between uploaded
metadata and retrieved metadata.

The primary criteria used in integration testing included end-to-end request
completion, correctness of inter-service outputs, validation of report and CID
data across steps, correct rejection of invalid requests, and preservation of
system boundaries across tenants and access levels.

\subsection{Performance Testing}

Performance testing focused on the responsiveness and stability of VERACISM
under expected verification traffic in the staging environment. The
performance assessment concentrated on the public CID verification endpoint
because this function is central to the use of the platform by students,
external verifiers, and partner institutions that require prompt integrity
checks.

The performance run was carried out using Postman Performance Testing against
the VERACISM API staging environment. The test profile used a fixed load with
100 virtual users over a duration of five minutes. The test scenario targeted
the valid public verification request and captured request volume,
throughput, response time, percentile response times, and observed
transport-level errors during the run.

The performance criteria for this assessment included sustained endpoint
availability during the test window, acceptable average and percentile
response times, stable throughput under concurrent traffic, and a minimal
error rate. The captured metrics will be retained as performance evidence for
later comparison and discussion in the results chapter.


\subsection{User Acceptance Testing}

User acceptance testing will concentrate on the main actors of VERACISM:
system administrators, tenant administrators, IT integrators, and external
verifiers using the public portal. Test scenarios will be derived from the
core functional flows and role-based capabilities of the system, including
dashboard review, report and file management, tenant administration, security
configuration, audit-log review, student and integrator management, public
verification, and API-based integration workflows.

The user acceptance assessment will use task-based scripts aligned with the
system acceptance criteria. Each UAT case will specify the participant role,
linked planned test case identifiers, the assigned task, the expected result,
and the final recorded outcome. Participants will perform the assigned tasks,
while observers will record task completion, time taken, defects
encountered, and any task blockers observed during execution. Results will be
recorded using objective completion outcomes such as \textit{Passed},
\textit{Partially Completed}, or \textit{Failed}, supported by notes and
evidence captured during the session.

\subsection{Security Testing}

Security evaluation for VERACISM will focus on protecting the
confidentiality, integrity, and availability of data in line with ISM
security policies. In particular, the system is expected to meet the
following security targets:

\begin{itemize}
  \item All network traffic uses HTTPS with current TLS versions;
  \item Passwords and secrets are never logged or exposed in client responses;
  \item Input validation and output encoding are applied to all external interfaces;
  \item The system is regularly scanned for vulnerabilities and no critical or high
        severity findings remain unresolved beyond agreed timelines.
\end{itemize}

VERACISM will be subjected to Vulnerability Assessment and Penetration Testing
(VAPT) before the initial go-live and after any significant change that may
affect its security posture. The target of the assessment is the VERACISM web
application and APIs, reachable under the staging URL
\texttt{https://sb-[update-domain-if-needed]} (to be confirmed). Test
credentials will be provisioned securely (for example, via encrypted email or
a password manager). Systems, subdomains, and third-party services not
explicitly included in the scope will remain out of scope.

The overall VAPT procedure for VERACISM will follow these steps:

\begin{enumerate}
  \item \textbf{Information gathering}  
        Collect details about infrastructure, technology stack, exposed
        services, and endpoints; define the attack surface for the web
        application and APIs.

  \item \textbf{Automated vulnerability scanning}  
        Run network and application scans using the tools in
        Table~\ref{table:security-testing-plan}. The objective is to identify:
        \begin{itemize}
          \item known CVEs in platforms and libraries;
          \item open ports and exposed services;
          \item misconfigurations and weak defaults;
          \item outdated software components;
          \item web vulnerabilities such as XSS, SQL injection, CSRF, and missing or
                insecure HTTP headers.
        \end{itemize}

  \item \textbf{Manual verification of findings}  
        Manually review automated results to eliminate false positives and to
        understand how each issue behaves on a running VERACISM instance.

  \item \textbf{Manual security testing}  
        Perform targeted tests on the affected flows and endpoints using safe,
        non-destructive payloads. For each confirmed finding, record severity,
        reproduction steps, payloads, and technical evidence (e.g., screenshots
        and logs).

  \item \textbf{Penetration testing focus areas}  
        Apply the OWASP Top 10 focus areas summarized in
        Table~\ref{table:owasp-focus} to ensure that access control, crypto,
        input handling, configuration, components, logging, and outbound calls
        are all tested in depth.

  \item \textbf{Remediation and re-testing}  
        Provide detailed technical findings and remediation guidance to the
        development and operations teams. After fixes are applied, perform
        re-tests to confirm that vulnerabilities are closed and that remediation
        has not introduced new security issues.

  \item \textbf{Reporting}  
        Produce a formal VAPT report containing all verified vulnerabilities,
        their severity and potential impact, reproduction details, and
        recommended remediation actions. Store reports and re-test results as
        evidence for audits and future reviews.
\end{enumerate}

Optionally, security assessment may be combined with load testing using Azure
Load Testing or an equivalent tool. In this mode, realistic user load will be
applied to VERACISM while security controls (rate limiting, throttling, and
brute-force protections) are observed under stress.

To keep VERACISM secure over time, continuous monitoring with dependency and
image scanning tools is recommended. Findings from these tools will be handled
under ISM's vulnerability management process.

Security acceptance criteria for the project are:

\begin{itemize}
  \item All Critical and High vulnerabilities identified in VAPT or automated
        scans must be fixed or formally risk-accepted by ISM Information
        Security before production release;
  \item Medium and Low issues must have a documented remediation plan and
        target date;
  \item VAPT reports, scan results, and re-test outcomes must be retained as
        evidence for audits and internal reviews.
\end{itemize}
