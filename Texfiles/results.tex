% Results and Discussion: VERACISM

\section{RESULTS AND DISCUSSION}

This section presents and discusses the outcomes of the four testing categories
conducted to evaluate the VERACISM platform: (1)~functional testing,
(2)~integration testing, (3)~performance testing, (4)~security testing, and
(5)~user acceptance testing.
Complete raw data for each category is reproduced in the appendices: detailed
assertion level tables appear in Appendices~\ref{appendix:integration-tests}
and \ref{appendix:functional-tests}; security findings in
Appendix~\ref{appendix:security}; performance configuration in
Appendix~\ref{appendix:performance}; and the full original test report PDFs
in Appendices~\ref{appendix:pdf-functional}
through~\ref{appendix:pdf-security-remediation}.

\subsubsection{Key functional findings}

The following system behaviours were confirmed by the combined testing programme:

\begin{enumerate}
  \item \textbf{Core pipeline is fully end to end functional.}  TC-E2E-001
    demonstrated that the complete upload to verify workflow: file submission,
    SHA-256 computation, threat scanning, Lighthouse anchoring, CID generation,
    and unauthenticated public verification: succeeds in a single automated
    run with \texttt{sha256Match: true} and \texttt{inLighthouse: true}.

  \item \textbf{Multi-engine threat detection gates function correctly.}
    TC-07A confirmed that a JPG embedding a webshell signature is flagged by
    the YARA engine (rule: \texttt{image\_webshell\_signature}) and blocked
    from Lighthouse upload, while ClamAV reports the same file as clean,
    demonstrating the value of the multi engine approach.

  \item \textbf{SHA-256 integrity is maintained end to end.}  The hash
    computed at upload (TC-ITO-001) is retrievable via the report endpoint
    (TC-ITO-003) and confirmed by the public verification endpoint
    (TC-E2E-001), covering the full document lifecycle.

  \item \textbf{Tenant data isolation prevents enumeration.}  TC-11 confirmed
    that a CID owned by a different tenant returns HTTP~404, not HTTP~403,
    preventing an authenticated attacker from inferring the existence of
    cross tenant documents.

  \item \textbf{Authentication and authorisation boundaries are correctly
    enforced.}  All protected endpoints rejected requests with invalid or
    missing tokens (HTTP~401/403).  The public verification endpoint returned
    HTTP~200 without any credentials (TC-EXT-001, TC-15).

  \item \textbf{Document immutability is enforced at the API level.}  No
    PUT or DELETE endpoints exist for report records (TC-REG-006); the API
    is append only by design, matching the tamper evident guarantee.

  \item \textbf{Tampered-document detection is reliable.}  A fabricated CID
    (TC-E2E-002) returned HTTP~404 with a descriptive message and no
    false positive verification result.
\end{enumerate}

\noindent Version~1.0 of the performance test revealed that the Nginx reverse proxy
rate limiting policy had been configured at a threshold below what
100~simultaneous virtual users required, causing the proxy to return
HTTP~429 (Too~Many~Requests) before requests ever reached the .NET~8
application server.  This was confirmed by disabling the rate limiting rules
for Version~1.1, after which the error rate dropped substantially.

This finding is operationally significant: the rate limiting layer is a
necessary security control (it protects against denial-of-service and
credential stuffing attacks), and it should \textit{not} be permanently
removed.  Rather, the threshold values must be tuned to permit the expected
peak concurrent-user volume while still blocking anomalously high request
rates.  The performance testing exercise itself demonstrated the value of
load testing under realistic concurrency conditions, surfacing a configuration
gap that would have directly impaired the user experience under production
load but would not have been detectable through functional or security testing
alone.

Full numeric metrics, including request counts, P50/P95 response time
percentiles, throughput graphs, and per second error-rate charts, are
reproduced verbatim in Appendix~\ref{appendix:pdf-perf-v10}
(Version~1.0) and Appendix~\ref{appendix:pdf-perf-v11} (Version~1.1).

\subsection{Overall Discussion}
% ----------------------------------------------------------

  Taken together, the five testing streams present a consistent picture of a
system that fulfils its core functional objectives while revealing
infrastructure level configuration gaps correctable before production
scale-out.

The \textbf{100\% automated assertion pass rate} across 182~assertions confirms
that the document verification pipeline, from upload and threat scanning
through to decentralised storage and public CID based verification, operates
as designed under normal conditions.  The 90.0\% full-pass rate on the
30-case manual suite reflects three bounded gaps in non-core subsystems
(retry logic, audit tooling, and verification logging) that do not compromise
the integrity or confidentiality of verified documents.

The performance testing exercise surfaced a real operational concern, the
rate limiting configuration, that would have impaired user experience under
production load had it not been identified in the testing phase.  The v1.0
error rate was caused entirely by infrastructure misconfiguration, not
application defects; v1.1 confirmed the application layer is capable of
sustaining 100-VU concurrent load.  The finding is correctable with
configuration changes alone.

The security assessment confirmed that the three most consequential risks
(cross tenant data access via BOLA) were identified, documented, and fully
remediated before the initial controlled deployment.  The DevSecOps workflow
embedded in the development process, combining automated DAST scanning
with manual code review, was effective at surfacing and closing
access-control gaps.  The absence of any confirmed Critical or High
severity vulnerability at deployment demonstrates a security baseline
appropriate for controlled institutional use.

One systemic observation across both security and functional testing phases
is that tenant-ownership checks are currently implemented as repeated inline
logic within individual controllers.  While functionally correct, this
pattern increases the risk of omission in future endpoint additions.
Centralising the checks into a shared base controller or a dedicated
ASP.NET~Core authorisation handler is identified as a recommended
architectural improvement for the next development phase.
